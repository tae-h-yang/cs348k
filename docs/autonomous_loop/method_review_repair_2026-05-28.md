# Method Review: Invalid Physical Reference Motions

Date: 2026-05-28.

## Question

Can we make MotionBricks-style humanoid references more physically reasonable
without claiming unsupported generator fine-tuning?

## Finding

The most defensible current method is a verifier-and-repair pipeline:

1. generate multiple stochastic MotionBricks proxy candidates,
2. select a low-risk candidate with the existing physical-awareness critic,
3. run a second-stage reference-conditioning repair that retimes and lightly
   smooths the selected reference,
4. score every repaired variant with the same critic,
5. keep the lowest-risk reference and render a before/after audit video.

This is closer to the MoVer idea than to training a new generator: explicit
predicates/metrics expose failures, then a repair/search loop tries constrained
edits. For humanoids the predicates are dynamics/contact/task checks rather
than SVG object-motion predicates.

## New Baseline

Script:

```bash
python scripts/repair_humanoid100_references.py --render
```

Inputs:

- `results/humanoid100_motionbricks_experiment/humanoid100_motionbricks_results.csv`
- selected K=8 qpos references generated by MotionBricks proxy calls

Outputs:

- `results/humanoid100_repaired_retimed/repair_summary.csv`
- `results/humanoid100_repaired_retimed/videos/*.mp4`
- `data/humanoid100_repaired_retimed/*.npy`

Repair variants:

- identity,
- retime 1.25x, 1.5x, 2.0x using root-quaternion Slerp,
- each retimed variant with light Gaussian or Savitzky-Golay smoothing.

Validation:

- qpos must be finite,
- qpos must have shape `(T, 36)`,
- dynamic repair requires `T >= 2`,
- root quaternions are normalized on copied data,
- near-zero quaternions are rejected with a clear `ValueError`.

## Result

On the 100-row forced-proxy MotionBricks experiment:

| Method | Mean risk |
|---|---:|
| K=1 baseline | 36.81 |
| K=8 best-of-K | 19.06 |
| K=8 plus retiming repair | 14.89 |

The repair improves 47/100 rows relative to K=8, worsens 0/100, and leaves
53/100 unchanged. The most common selected variant is identity, which is good:
the repair does not rewrite motions unless the score says there is a dynamics
benefit.

## Limits

- Retiming changes duration and perceived tempo.
- This is not MotionBricks fine-tuning.
- This is not a full physical optimizer with contact constraints.
- This is not a semantic prompt-following fix: 78/100 rows are still forced
  nearest-mode proxies because the local MotionBricks preview does not expose
  arbitrary text-conditioned generation for those tasks.
- SONIC/native-controller validation remains the stronger acceptance gate for
  executable references.

## Next Better Method

The next meaningful step is controller-aware repair instead of critic-only
repair:

1. keep the retiming/smoothing search as the cheap proposal distribution,
2. export the best few variants to native SONIC,
3. select by survival, root drift, contact sanity, and visual audit,
4. train a trajectory-level acceptance model only after native labels exist.

This would be a real controller-in-the-loop curation method. Training a
large-looking model without native acceptance labels would not address the
user-visible failure mode.
